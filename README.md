# 广东省干部培训网络学院专题学习助手

本目录中的 `gdgbpx-workshop-helper.user.js` 是针对新版“专题学习”Vue 页面重新编写的独立 Tampermonkey 脚本，不依赖三份旧脚本。

## 使用方式

1. 在 Tampermonkey 中新建脚本，粘贴 `gdgbpx-workshop-helper.user.js` 的全部内容并保存。
2. 停用针对同一网站的旧脚本，避免多个脚本同时点击或刷新。
3. 用户自行登录并完成课程报名。
4. 手动进入“专题学习 → 在学”。
5. 在左下角悬浮面板点击“开始”；可拖动标题栏移动面板。脚本固定使用正常 `1×` 速度以保证服务器记满学习时长。

脚本支持开始、暂停、继续、停止、跳过当前课程、重新检查进度、静音、自动恢复普通暂停和分页处理。控制面板默认贴在左下角，可拖动标题栏移动，并会记住位置。

## 诊断日志

脚本会把主站、播放器外层页面和播放器 iframe 的日志汇总保存，最多保留最近 800 条。日志包括：

- 脚本启动、页面地址和运行状态变化；
- 专题与课程列表的读取结果、点击和分页；
- 视频源、`readyState`、`networkState`、时长和播放进度；
- 元数据加载、开始播放、暂停、缓冲、卡死、播放失败和视频错误；
- 播放器窗口关闭、专题刷新以及未捕获的 JavaScript 异常。

面板底部提供三个操作：

- **复制**：把当前状态和全部日志复制到剪贴板；
- **下载**：生成一个 `gdgbpx-helper-log-日期.json` 文件；
- **清空**：删除旧日志，方便重新复现问题。

出现问题时，先不要刷新或清空日志，点击“下载”，然后把生成的 JSON 文件发来即可。脚本会自动遮盖常见的 Token、Cookie、授权字段、会话参数和用户标识；分享前仍建议自行快速检查一遍文件内容。

Tampermonkey 菜单中也提供“复制诊断日志 / 下载诊断日志 / 清空诊断日志”，面板被隐藏时仍可使用。

### 自动保存到本机供 Codex 检查

双击运行项目中的 `start-debug-bridge.cmd` 后，脚本会把已经脱敏的日志自动发送到：

`logs/gdgbpx-debug.ndjson`

调试桥只监听 `127.0.0.1:17891`，不接受局域网或互联网连接，并要求 Tampermonkey 请求携带专用请求头。连接失败不会影响课程播放，日志仍会暂存在 Tampermonkey 中；恢复连接后会补发最近等待发送的记录。

面板显示“本机调试桥：已连接，日志自动保存”后，出现问题只需告诉 Codex大致发生时间和现象，Codex 可以直接读取项目日志，不需要再复制或下载文件。

### v1.2.1 修复

播放器事件现在绑定到打开它的具体课程。上一节播放器窗口延迟发出的“已关闭”等事件，如果此时已经进入下一课，会被识别为过期事件并忽略，避免下一课刚启动就被旧事件打断。

### v1.2.2 修复

无播放进度默认只告警并继续尝试恢复，不再自动关闭播放器。面板新增“卡死后自动重开”开关，只有用户明确开启后，达到设定的无进度时间才会关闭并重新打开课程。这可避免站点短暂暂停或异常缓冲时被误判为必须关闭。

### v1.2.3 修复

视频原生 `ended` 或网站结束提示出现后，脚本等待 4 秒供站点保存完成记录，然后明确关闭播放器窗口，不再完全依赖网站的自动关闭行为。父页面随后刷新进度并进入下一课。

### v1.2.4 修复

脚本只把 `classType=3` 识别为“在学”列表，不再把“进行中”等其他专题列表的临时空内容误判为全部完成。“在学”列表没有课程卡片时还会等待至少 8 秒并确认加载状态，避免 Vue 页面切换期间的空 DOM 导致播放器收到错误的 `complete` 状态而暂停。

### v1.3.0 计时策略修复

实测确认服务器按实际学习时长而非单纯的视频播放位置累计课程进度，因此脚本固定使用 `1×`。如果旧版倍速已经把播放器记忆位置写到结尾、但服务器仍显示未完成，新版会把视频回退到服务器已确认进度附近并保留 30 秒重叠，以正常速度补足剩余学习时间。所有主动关闭播放器的路径现在都会记录明确的关闭原因。

### v1.3.1 防循环修复

播放器即便在加载完成后才被网站跳到结尾，原生 `ended` 事件也会再次检查服务器已确认进度；未完成时立即回退继续，不会关闭。播放器连续三次未能启动时，脚本会暂停并提示检查浏览器弹窗权限，而不是无限点击课程入口。

### v1.5.3 实时完成状态监控

播放过程中不再刷新用户当前看到的专题详情页。脚本会在同源后台 iframe 中每约 6.5 秒重新读取一次专题详情，并在 Vue DOM 尚未渲染完成时继续重试读取；只把当前课程的明确状态文本 `已完成` 作为关闭播放器和进入下一节的条件；百分比仅用于面板显示和诊断。如果视频结束或播放器提前关闭但服务器仍未标记 `已完成`，脚本会留在当前课程并继续复查，不会直接跳到下一节。

## 播放与进度原则

- 固定使用 `1×`。实测站点按实际学习时长累计进度，`1.5× / 2×` 会出现视频已结束但课程进度未满的情况。
- 不向前拖动进度、不调用或伪造完成接口、不修改课程标识和心跳数据；仅在旧版倍速留下“播放器已在结尾、服务器仍未完成”的异常状态下，回退到服务器已确认位置附近继续正常播放。
- 正常等待视频 `ended`；“卡死重试”仅在播放器持续数分钟没有任何时间进度时关闭播放器并重新检查，不是固定播放三分钟就关闭。
- 如果课程出现内嵌提问，脚本暂停自动恢复并提示用户手动作答。
- 自动播放会等待站点装载有效视频源和元数据，再点击 Video.js 的播放按钮；不会在空播放器阶段反复调用 `play()`。

## 已确认的网站行为

- 播放器每 30 秒更新一次 `lesson_location` 和实际播放 `session_time`。
- 外层页面每 60 秒向服务器发送一次心跳。
- 视频触发 `ended` 后，站点调用自身完成接口；可通过后台同源详情探测页读取最新的 `已完成` 状态，不打断正在播放的窗口。

## 首次实测重点

- 浏览器是否拦截脚本点击后产生的播放器窗口。
- 视频结束后，日志中是否出现 `server-status-probe-result`，并且只有读取到 `status=已完成` 后才关闭播放器、进入下一课。
- “暂停/继续/停止/跳过当前”能否跨播放器窗口生效。
- 课程全部完成后，专题是否从“在学”移到“已学”，脚本是否继续下一专题。
